## Overview

Both PDF-to-PDF and PDF-to-HTML workflows produce accessibility-enhanced outputs that meet WCAG 2.1 Level AA standards. This guide explains what improvements are made, how to verify compliance, and how to interpret the results.

## PDF-to-PDF Results

### Output Files

The PDF-to-PDF workflow produces:

```
result/
└── COMPLIANT_<filename>.pdf

temp/<filename>/accessability-report/
├── <filename>_accessibility_report_before_remediation.json
└── <filename>_accessibility_report_after_remediation.json
```

#### `COMPLIANT_<filename>.pdf`

Your remediated PDF with the following enhancements:

**Document Structure:**
- PDF tags for logical reading order
- Semantic structure tree (headings, paragraphs, lists)
- Language metadata (auto-detected)
- Document title in metadata
- Viewer preferences (display title in window)

**Accessibility Features:**
- Tagged content for screen readers
- Alt text for all images (AI-generated with context)
- Table of contents based on headings
- Proper heading hierarchy (H1-H6)
- Table structure with headers
- Form field labels and descriptions
- List structures (ordered and unordered)

**Image Accessibility:**
- Descriptive alt text generated by Amazon Bedrock Nova Pro
- Context-aware descriptions (considers surrounding text)
- Decorative images marked as artifacts
- Figure captions preserved

#### Accessibility Reports (JSON)

**Before Remediation Report:**
Documents all accessibility issues in the original PDF:

```json
{
  "compliance_status": "non_compliant",
  "issues": [
    {
      "type": "Missing alt text",
      "severity": "critical",
      "count": 15
    },
    {
      "type": "Untagged content",
      "severity": "major",
      "count": 42
    }
  ],
  "wcag_criteria": {
    "passed": 12,
    "failed": 8
  }
}
```

**After Remediation Report:**
Shows improvements and remaining issues (if any):

```json
{
  "compliance_status": "compliant",
  "issues": [
    {
      "type": "Missing alt text",
      "severity": "critical",
      "count": 0,
      "remediated": 15
    }
  ],
  "wcag_criteria": {
    "passed": 20,
    "failed": 0
  }
}
```

<Tip>
  Compare the before and after reports to see exactly what accessibility improvements were made to your PDF.
</Tip>

## PDF-to-HTML Results

### Output Package

The PDF-to-HTML workflow produces a comprehensive package:

```
final_<filename>.zip
├── remediated_html/
│   ├── index.html              # Main accessible HTML
│   ├── styles.css
│   └── images/
│       ├── image_1.png
│       └── ...
├── remediation_report.html     # Visual compliance report
└── usage_data.json            # Processing metrics
```

#### `remediated_html/index.html`

The primary output - a fully accessible HTML document:

**Document Structure:**
```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Document Title</title>
</head>
<body>
  <!-- Properly structured semantic HTML -->
</body>
</html>
```

**Accessibility Features Applied:**

1. **Semantic HTML Elements**
   ```html
   <header>, <nav>, <main>, <article>, <aside>, <footer>
   ```

2. **ARIA Landmarks**
   ```html
   <nav role="navigation" aria-label="Main navigation">
   <main role="main">
   <aside role="complementary">
   ```

3. **Hierarchical Headings**
   ```html
   <h1>Document Title</h1>
     <h2>Section 1</h2>
       <h3>Subsection 1.1</h3>
     <h2>Section 2</h2>
   ```

4. **Image Alt Text**
   ```html
   <img src="images/image_1.png" 
        alt="Bar chart showing quarterly revenue growth of 15% from Q1 to Q4">
   ```

5. **Accessible Tables**
   ```html
   <table>
     <caption>Quarterly Sales Data</caption>
     <thead>
       <tr>
         <th scope="col">Quarter</th>
         <th scope="col">Revenue</th>
       </tr>
     </thead>
     <tbody>
       <tr>
         <th scope="row">Q1</th>
         <td>$1.2M</td>
       </tr>
     </tbody>
   </table>
   ```

6. **Form Labels**
   ```html
   <label for="name">Full Name:</label>
   <input type="text" id="name" name="name" required>
   ```

7. **Link Context**
   ```html
   <a href="#section1">Read more about our services</a>
   <!-- Not just "Click here" -->
   ```

8. **Color Contrast**
   - Minimum 4.5:1 for normal text
   - Minimum 3:1 for large text
   - Compliant color schemes applied

#### `remediation_report.html`

Visual report showing all accessibility improvements:

**Report Sections:**

1. **Summary Statistics**
   - Total issues identified
   - Issues remediated
   - Compliance status
   - Processing time

2. **Issues by Category**
   - Images (alt text)
   - Document structure
   - Headings
   - Links
   - Tables
   - Forms
   - Color contrast
   - Landmarks

3. **Severity Breakdown**
   - Critical issues fixed
   - Major issues fixed
   - Minor issues fixed

4. **WCAG Success Criteria**
   - List of criteria met
   - Before/after comparison
   - Compliance level achieved

5. **Detailed Issue List**
   Each issue shows:
   - Type and severity
   - Original HTML selector
   - Remediation applied
   - Status (remediated/failed)

**Example Report Entry:**
```
Issue: Missing alt text
Severity: Critical
Element: <img src="chart.png">
Remediation: Added alt="Sales trend chart showing 20% growth"
Status: ✓ Remediated
WCAG Criteria: 1.1.1 Non-text Content (Level A)
```

<Tip>
  Open `remediation_report.html` in a browser to see a visual representation of all accessibility improvements.
</Tip>

## Accessibility Improvements Made

### WCAG 2.1 Level AA Compliance

Both workflows address these WCAG success criteria:

#### Perceivable (Principle 1)

**1.1.1 Non-text Content (Level A)**
- ✓ Alt text for all images
- ✓ Decorative images properly marked
- ✓ Complex images with detailed descriptions

**1.3.1 Info and Relationships (Level A)**
- ✓ Semantic HTML structure
- ✓ Proper heading hierarchy
- ✓ Table headers and structure
- ✓ Form label associations

**1.3.2 Meaningful Sequence (Level A)**
- ✓ Logical reading order
- ✓ Content flow preserved from PDF

**1.4.3 Contrast (Level AA)**
- ✓ 4.5:1 contrast for normal text
- ✓ 3:1 contrast for large text
- ✓ Automatic contrast adjustments

#### Operable (Principle 2)

**2.4.1 Bypass Blocks (Level A)**
- ✓ Skip navigation links
- ✓ ARIA landmarks

**2.4.2 Page Titled (Level A)**
- ✓ Descriptive page titles
- ✓ Document metadata

**2.4.4 Link Purpose (Level A)**
- ✓ Descriptive link text
- ✓ Context for links

**2.4.6 Headings and Labels (Level AA)**
- ✓ Descriptive headings
- ✓ Clear form labels

#### Understandable (Principle 3)

**3.1.1 Language of Page (Level A)**
- ✓ Language attribute set
- ✓ Auto-detection of language

**3.1.2 Language of Parts (Level AA)**
- ✓ Language changes marked

**3.2.4 Consistent Identification (Level AA)**
- ✓ Consistent component behavior
- ✓ Predictable navigation

#### Robust (Principle 4)

**4.1.2 Name, Role, Value (Level A)**
- ✓ ARIA labels and roles
- ✓ Form control states
- ✓ Semantic elements

## Verifying WCAG Compliance

### Automated Testing Tools

Use these tools to verify accessibility:

**Browser Extensions:**
- **WAVE** (WebAIM) - Visual feedback on accessibility issues
- **axe DevTools** - Detailed WCAG compliance testing
- **Lighthouse** (Chrome DevTools) - Accessibility audit scores

**Screen Readers:**
- **NVDA** (Windows) - Free screen reader testing
- **JAWS** (Windows) - Professional screen reader
- **VoiceOver** (macOS) - Built-in screen reader

### Manual Testing

**Keyboard Navigation:**
```
1. Tab through all interactive elements
2. Verify focus indicators are visible
3. Check for keyboard traps
4. Test form submission
```

**Screen Reader Testing:**
```
1. Navigate by headings
2. Listen to image descriptions
3. Navigate through tables
4. Test form labels
5. Verify landmark regions
```

**Visual Inspection:**
```
1. Check color contrast
2. Verify text sizing
3. Test at 200% zoom
4. Review heading hierarchy
5. Validate link text
```

## Quality Assurance Checklist

After receiving your remediated outputs:

<Steps>
  <Step title="Download and Extract">
    - ✓ Download all result files
    - ✓ Extract ZIP packages
    - ✓ Verify file integrity
  </Step>

  <Step title="Visual Review">
    - ✓ Open in browser/PDF reader
    - ✓ Check layout preservation
    - ✓ Verify images display correctly
    - ✓ Review text formatting
  </Step>

  <Step title="Accessibility Testing">
    - ✓ Run WAVE or axe DevTools
    - ✓ Test keyboard navigation
    - ✓ Use screen reader
    - ✓ Check color contrast
  </Step>

  <Step title="Review Reports">
    - ✓ Read remediation report
    - ✓ Verify all issues addressed
    - ✓ Check WCAG compliance level
    - ✓ Note any remaining manual fixes
  </Step>

  <Step title="Production Deployment">
    - ✓ Test in target environment
    - ✓ Validate with users
    - ✓ Document any customizations
    - ✓ Establish maintenance plan
  </Step>
</Steps>

## Best Practices

1. **Always Review Reports** - Don't skip the accessibility reports; they show exactly what was fixed
2. **Test with Real Users** - Automated tools catch ~70% of issues; test with actual assistive technology users
3. **Validate Context** - Check that AI-generated alt text accurately describes images in context
4. **Maintain Source** - Keep original PDFs for reference and potential reprocessing
5. **Regular Testing** - Periodically re-test remediated content with updated tools
6. **Manual Refinements** - Some context-specific improvements may require manual editing
