The PDF-to-PDF remediation solution uses Adobe PDF Services API for automated tagging and structure detection. This page explains how to obtain Adobe credentials and configure them in AWS Secrets Manager.

<Note>
Adobe credentials are **only required for the PDF-to-PDF solution**. The PDF-to-HTML solution uses AWS Bedrock Data Automation instead.
</Note>

## Overview

Adobe PDF Services API credentials consist of two components:

1. **Client ID** — Public identifier for your Adobe API project
2. **Client Secret** — Private authentication key (keep secure)

These credentials are stored securely in AWS Secrets Manager and accessed by ECS tasks and Lambda functions during PDF processing.

## Obtaining Adobe Credentials

### Step 1: Create an Adobe Developer Account

1. Go to the [Adobe Developer Console](https://developer.adobe.com/console)
2. Click **Sign In** (or **Sign Up** if you don't have an account)
3. Use your Adobe ID or create a new one

<Tip>
If your organization already has an Adobe Developer account, check with your admin before creating a new one.
</Tip>

### Step 2: Create a New Project

1. In the Adobe Developer Console, click **Create new project**
2. Choose **Add API**
3. Select **PDF Services API** from the list
4. Click **Next**

### Step 3: Configure API Credentials

1. Choose **OAuth Server-to-Server** as the credential type
2. Configure the OAuth scopes (select all available scopes)
3. Click **Save configured API**

### Step 4: Retrieve Your Credentials

1. In your project, click on **PDF Services API** in the left sidebar
2. Navigate to the **Credentials** tab
3. You'll see two values:
   - **Client ID** — Copy this value
   - **Client Secret** — Click **Retrieve client secret** and copy this value

<Warning>
The Client Secret is only shown once. Save it securely. If you lose it, you'll need to regenerate a new secret.
</Warning>

## Storing Credentials in AWS Secrets Manager

### Automatic Setup (via Deployment Script)

The deployment script automatically creates the secret during deployment:

<CodeGroup>

```bash deploy.sh
./deploy.sh
```

</CodeGroup>

You'll be prompted to enter your Adobe credentials:

```
Adobe PDF Services API credentials are required:
(These will be stored securely in AWS Secrets Manager)
   Enter Adobe API Client ID: <paste your Client ID>
   Client ID received! ✅
   Enter Adobe API Client Secret: <paste your Client Secret>
   Client Secret received! ✅
```

The script creates the secret at the path `/myapp/client_credentials` with the correct structure.

### Manual Setup (via AWS CLI)

If you need to create or update the secret manually:

<CodeGroup>

```bash Create Secret
aws secretsmanager create-secret \
  --name /myapp/client_credentials \
  --description "Adobe PDF Services API credentials" \
  --secret-string '{
    "client_credentials": {
      "PDF_SERVICES_CLIENT_ID": "your-client-id-here",
      "PDF_SERVICES_CLIENT_SECRET": "your-client-secret-here"
    }
  }'
```

```bash Update Existing Secret
aws secretsmanager update-secret \
  --secret-id /myapp/client_credentials \
  --secret-string '{
    "client_credentials": {
      "PDF_SERVICES_CLIENT_ID": "your-new-client-id",
      "PDF_SERVICES_CLIENT_SECRET": "your-new-client-secret"
    }
  }'
```

</CodeGroup>

### Secret Structure

The secret must follow this exact JSON structure:

<CodeGroup>

```json Secret Format
{
  "client_credentials": {
    "PDF_SERVICES_CLIENT_ID": "<Your Adobe Client ID>",
    "PDF_SERVICES_CLIENT_SECRET": "<Your Adobe Client Secret>"
  }
}
```

</CodeGroup>

<ParamField path="client_credentials.PDF_SERVICES_CLIENT_ID" type="string" required>
  Your Adobe PDF Services API Client ID from the Developer Console.
</ParamField>

<ParamField path="client_credentials.PDF_SERVICES_CLIENT_SECRET" type="string" required>
  Your Adobe PDF Services API Client Secret from the Developer Console.
</ParamField>

<Warning>
The secret **must** be stored at the path `/myapp/client_credentials`. Do not change this path without updating the IAM policies in `app.py`.
</Warning>

## Verifying the Secret

After creating the secret, verify it's configured correctly:

<CodeGroup>

```bash Retrieve Secret Value
aws secretsmanager get-secret-value \
  --secret-id /myapp/client_credentials \
  --query SecretString \
  --output text | jq .
```

</CodeGroup>

Expected output:

```json
{
  "client_credentials": {
    "PDF_SERVICES_CLIENT_ID": "your-client-id",
    "PDF_SERVICES_CLIENT_SECRET": "your-client-secret"
  }
}
```

## Testing Adobe Integration

### Test with a Sample PDF

After deployment, test the Adobe integration:

1. Upload a test PDF to the S3 bucket under the `pdf/` prefix:

<CodeGroup>

```bash Upload Test File
aws s3 cp test.pdf s3://your-bucket-name/pdf/test.pdf
```

</CodeGroup>

2. Monitor the Step Functions execution in the AWS Console
3. Check CloudWatch Logs for the Adobe AutoTag ECS task:

<CodeGroup>

```bash View ECS Logs
aws logs tail /ecs/pdf-remediation/adobe-autotag --follow
```

</CodeGroup>

4. Look for successful Adobe API calls:

```
[INFO] Successfully authenticated with Adobe PDF Services
[INFO] Processing PDF with AutoTag
[INFO] AutoTag completed successfully
```

### Common Test Errors

<AccordionGroup>

<Accordion title="401 Unauthorized - Invalid Client Credentials">

**Cause:** Incorrect Client ID or Client Secret.

**Solution:**
1. Verify your credentials in the Adobe Developer Console
2. Update the secret in Secrets Manager
3. Redeploy or restart the ECS task

</Accordion>

<Accordion title="403 Forbidden - API Access Denied">

**Cause:** Your Adobe account doesn't have permission to use PDF Services API.

**Solution:**
1. Verify PDF Services API is enabled in your Adobe project
2. Check that OAuth scopes are correctly configured
3. Ensure your trial hasn't expired (if using trial account)

</Accordion>

<Accordion title="429 Too Many Requests - Rate Limit Exceeded">

**Cause:** You've exceeded your monthly transaction limit (1,000 for trial accounts).

**Solution:**
1. Wait for the monthly limit to reset
2. Upgrade to an enterprise account for higher limits
3. Contact Adobe support for quota increase

</Accordion>

</AccordionGroup>

## Rotating Credentials

For security compliance, periodically rotate your Adobe credentials:

### Step 1: Generate New Credentials

1. Go to the Adobe Developer Console
2. Navigate to your project → **PDF Services API** → **Credentials**
3. Click **Add credential** to create a new OAuth Server-to-Server credential
4. Copy the new Client ID and Client Secret

### Step 2: Update Secrets Manager

<CodeGroup>

```bash Update Secret
aws secretsmanager update-secret \
  --secret-id /myapp/client_credentials \
  --secret-string '{
    "client_credentials": {
      "PDF_SERVICES_CLIENT_ID": "new-client-id",
      "PDF_SERVICES_CLIENT_SECRET": "new-client-secret"
    }
  }'
```

</CodeGroup>

### Step 3: Restart Services

ECS tasks will automatically pick up the new credentials on their next execution. To force an immediate update:

1. Stop any running ECS tasks (they'll restart automatically)
2. Or wait for the next PDF processing job to use the new credentials

### Step 4: Delete Old Credentials

After verifying the new credentials work:

1. Return to the Adobe Developer Console
2. Delete the old OAuth credential
3. This prevents unauthorized use of the old secret

<Tip>
Schedule credential rotation every 90 days for security best practices.
</Tip>

## IAM Permissions for Secrets Manager

The following IAM permissions are automatically configured for secret access:

**ECS Task Role:**
```json
{
    "Sid": "SecretsManagerAccess",
    "Effect": "Allow",
    "Action": ["secretsmanager:GetSecretValue"],
    "Resource": "arn:aws:secretsmanager:${Region}:${AccountId}:secret:/myapp/*"
}
```

**Configuration Location:** `app.py:129`

**Lambda Accessibility Checker Roles:**
```json
{
    "Sid": "SecretsManagerAccess",
    "Effect": "Allow",
    "Action": ["secretsmanager:GetSecretValue"],
    "Resource": "arn:aws:secretsmanager:${Region}:${AccountId}:secret:/myapp/*"
}
```

**Configuration Location:** `app.py:322`, `app.py:348`

## Security Best Practices

<Warning>
**Never commit credentials to source control.** Always use AWS Secrets Manager or environment variables.
</Warning>

### Do's ✅

- Store credentials in AWS Secrets Manager
- Use IAM policies to restrict secret access
- Rotate credentials regularly (every 90 days)
- Monitor Adobe API usage in CloudWatch
- Use separate Adobe accounts for dev/staging/production

### Don'ts ❌

- Hard-code credentials in Lambda functions or containers
- Share Client Secrets via email or chat
- Commit `.env` files with credentials
- Use production credentials in development environments
- Store credentials in S3 buckets (even private ones)
